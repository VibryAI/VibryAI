# VibryAI 产品需求

> 本文档是 VibryAI Server、VibryCard 和外部 Agent 接入的产品需求基线。
> 工程架构与迁移细节见 `VibryServer_Refactor_Plan.md`。

## 1. 产品目标

Vibry.AI 是单用户的个人认知内核。它持续吸收用户的录音、对话、文档和
Agent 工作记录，在保留原始证据的前提下形成项目记忆、项目洞察和全局认知，
并通过 VibryCard、Dashboard、OpenAI API 和 MCP 提供一致的智能能力。

核心体验不是“保存了很多资料”，而是用户能够在前端持续感知到：

1. AI 知道当前录音、当前项目和用户长期背景；
2. AI 能跨多次录音和对话发现变化、矛盾、风险与机会；
3. AI 给出的分析和建议与用户的目标、偏好和历史决策一致；
4. 用户在自然对话中的修正、确认和选择会改变后续回答；
5. 所有重要判断都可以追溯到原始录音、聊天或文档证据。

## 2. 产品边界

### 2.1 VibryAI Server

唯一的认知内核和数据事实源，负责：

- 大模型聚合及 OpenAI 兼容 API；
- 录音转写、摘要和认知处理；
- L0-L4 记忆与项目洞察；
- 项目管理、全局对话和 Dashboard；
- 文档、VibryCard、Agent 和插件接入；
- 任务和提醒数据接口；
- 用户反馈、修正、版本和证据管理。

当前产品为单用户版，认知所有者固定为 `admin`。

### 2.2 VibryCard

现实世界的主要采集端和移动体验入口，负责：

- 录音、上传、播放和转写状态展示；
- 展示单次录音报告及个性化建议；
- 在录音上下文中与 VibryAI 对话和纠错；
- 展示项目、任务和提醒；
- 不在客户端维护独立的长期记忆。

### 2.3 外部 Agent 和插件

Claude Code、Zcode、未来桌面端和其他 Agent 都是 VibryAI 的外部客户端。
它们通过 REST、OpenAI API、MCP 或同步插件访问 Server，不直接读写认知数据库。

插件同步的会话和工作记录以文件或结构化 Source 输入；同步过程本身写入项目日志。

## 3. 统一认知模型

### 3.1 L0：原始素材

L0 保存不可丢失、可追溯的原始输入：

- VibryCard 录音、转写和结构化纪要；
- 录音详情、项目和全局范围内的用户聊天；
- 用户上传的 Word、PDF、表格、图片、文本等文件；
- Claude Code、Zcode 等 Agent 的会话、任务结果和产物；
- 手工笔记和未来外部系统同步内容。

每条 Source 必须包含稳定外部 ID、来源、时间、内容哈希、原始文件位置、处理状态和
项目归属。一个 Source 可以关联多个项目。

### 3.2 L1：可验证记忆

L1 将 L0 转为可检索、可修正的主张。每条主张至少包含：

- 内容与主张类型：事实、目标、决策、偏好、行动、风险、约束等；
- 四网络归属：`world`、`experience`、`observation`、`opinion`；
- 实体、发生时间和有效时间；
- 置信度、状态和版本；
- 原始证据片段及 Source 引用；
- 支持、反对、相似、矛盾、替代等关系。

用户明确表达的事实、目标、修正、偏好和确认可以进入 L1。AI 的回答或推测不能自动
成为用户事实，除非用户确认，或者存在足够的外部证据。

### 3.3 L2：项目上下文

项目是多来源认知的主要组织单位，而不是一种素材类型。项目由用户维护，AI 负责推荐
归类。一个项目至少包含：

- 名称、背景信息、Goal、阶段、状态和标签；
- 对话、文档、录音、任务和日志；
- 项目主张、约束、未决问题和历史决策；
- 多个版本的 L3 项目报告。

一个 Source 可以属于多个项目。用户指定的项目归属优先于 AI 推断；未能确定归属的
Source 进入待归类素材箱。

### 3.4 L3：项目级报告和洞察

单个录音报告属于 L0/L1 的展示结果。L3 是按项目聚合的每日或按需报告，综合项目下的
录音、对话、文档、Agent 记录、任务、历史主张和上一版报告。

L3 至少包含：

- 今日新增事实、决策和进展；
- Goal 进展和关键指标变化；
- 风险、矛盾、信息缺口和未决问题；
- 商业机会和跨素材模式；
- 下一步建议、行动项和责任人；
- 相比上一版报告发生的变化；
- 每项结论对应的证据。

夜间只重新处理存在新增素材、用户反馈或状态变化的项目。用户也可以手动触发重算。

### 3.5 L4：全局个人认知

L4 是跨项目、跨时间形成的全局判断，包括：

- 长期目标、角色和战略重心；
- 专业领域、能力边界和资源条件；
- 决策偏好、判断原则和工作方式；
- 反复出现的优势、盲点和风险模式；
- 对行业、产品、客户和商业化的长期观点；
- 哪些证据或条件会促使用户改变判断。

L4 采用候选认知机制：AI 发现模式、展示证据、向用户提出少量确认题，用户确认、修改
或否定后形成版本。L4 必须低频、渐进、可解释、可撤销，不因一次聊天直接改变。

## 4. 三种对话范围

所有对话都必须先检索对应范围的记忆，再生成回答；对话结束后异步识别用户是否提供了
事实、目标、修正、确认、偏好、任务或项目操作。

### 4.1 录音详情对话

上下文优先级：当前录音原文和纪要 > 当前录音主张 > 所属项目记忆 > 必要的 L4 认知。

用户可以追问录音内容、要求分析、补充背景、修正转写或事实、创建任务。修正需要更新
底层主张和证据，并触发相关项目 L3 重新计算，而不是只修改聊天回答。

### 4.2 项目对话

上下文优先级：项目背景与 Goal > 项目有效主张 > 项目文档、录音和任务 > L3 > 必要的
L4 认知。

项目对话用于分析项目、更新目标与约束、补充事实、修正报告、创建任务和调整项目状态。
项目聊天中的长期内容写回项目记忆；临时提问和寒暄不进入长期记忆。

### 4.3 VibryAI 全局对话

全局对话可以检索全部项目和 L4，但应根据问题自动缩小范围，避免无关素材污染上下文。
它用于跨项目分析、每日总览、战略讨论和 L4 认知校准。

全局聊天中的明确修正可以影响 L1-L4；涉及 L4 的修改必须进入候选、确认和版本流程。

## 5. 隐式反馈闭环

用户不需要进入专门后台维护记忆。录音详情、项目和全局聊天本身就是主要反馈入口。

每个用户输入需要识别为以下一种或多种意图：

- 普通提问：只检索和回答，不写长期记忆；
- 补充事实：创建或补充 L1 主张和证据；
- 修正：标记原主张错误或过时，建立替代关系；
- 确认或否定：调整候选结论状态和置信度；
- 目标或约束：更新项目 Goal、项目记忆或 L4 候选；
- 偏好：按适用范围进入项目级或全局 opinion；
- 创建任务：形成任务并保留来源消息；
- 项目操作：归类、移动、合并、拆分或归档。

反馈事件必须记录作用范围、目标对象、原始消息、处理动作、前后版本和时间。用户的明确
修正具有最高优先级，但仍保留旧版本以便追溯。

## 6. 项目工作区

“项目管理”是 Dashboard 一级菜单，由用户维护。项目详情采用固定工作区：

1. **概览**：项目背景、Goal、阶段、状态、最新 L3 和待确认事项；
2. **对话**：调用项目记忆的专属聊天；
3. **文档**：用户上传及插件同步的文件和结构化记录；
4. **录音**：VibryCard 写入并关联到项目的录音；
5. **任务**：由聊天或录音行动项创建的任务；
6. **日志**：插件同步、文件处理、认知处理和用户操作日志。

日志默认不作为长期记忆。只有成功结果、失败原因、重要变更和用户确认的高价值操作才
提取为 L1 事实或经验。

## 7. 任务与提醒

用户可以在三种聊天范围内自然创建任务。任务必须包含项目、标题、截止时间、状态、来源
消息、来源录音或文档以及提醒设置。

缺少截止时间时，AI 应在不打断主要对话的情况下提出简短确认。任务通过统一 API 提供给
未来 VibryAI App，由 App 承担本地通知和提醒展示。

## 8. 检索和回答原则

- 回答前先判断录音、项目或全局范围；
- 使用语义检索、实体、时间和项目关系共同召回；
- 优先使用用户确认、最新且证据充分的主张；
- 明确区分事实、用户观点和 AI 推断；
- 对关键结论提供可打开的录音、聊天或文档证据；
- 遇到矛盾时展示冲突并询问，不擅自选择有利版本；
- 控制上下文预算，不把项目全部原文塞入模型；
- 项目聊天必须调用项目记忆，全局聊天按需调用项目与 L4 记忆。

## 9. 处理节奏

- 实时：L0 入库、转写、L1 提取、项目推荐、聊天检索和明确修正；
- 夜间：去重、矛盾与替代识别、项目归类更新、脏项目 L3 重算；
- 每周或证据阈值触发：跨项目模式发现和 L4 候选生成；
- 用户反馈后：立即更新相关状态，并异步重算受影响的 L3/L4 候选。

## 10. 隐私与可靠性

- 所有认知数据固定属于单用户 `admin`；
- 外部插件不得直接访问数据库；
- 原始 Source 和人工修正不可静默删除；
- 所有派生结论必须保留模型、提示版本、证据和生成时间；
- 删除项目只删除项目关系，原始素材是否删除必须单独确认；
- 模型或向量服务不可用时任务明确失败并可重试，不伪造成功结果。

## 11. 版本边界

### v0.3：认知主链路候选版

- VibryCard 录音、转写和纪要；
- L0、L1、基础 L2 和 L3；
- FastEmbed 检索、单用户 Dashboard、OpenAI API 和 MCP；
- 不承诺完整项目工作区和 L4。

### v0.4：项目工作台

- 项目管理一级菜单；
- 背景、Goal、对话、文档、录音、任务和日志；
- 三种对话范围及隐式反馈；
- 项目自动推荐、多项目归属和 L3 每日报告。

### v0.5：全局认知

- L4 候选、确认、否定、修正和版本；
- 跨项目认知校准对话；
- L4 对项目和全局回答的受控注入；
- VibryAI App 所需的任务和提醒 API。

## 12. 核心验收标准

1. 用户在录音详情追问时，回答能够引用当前录音并结合所属项目背景；
2. 用户修正录音中的事实后，后续录音、项目和全局回答均使用新版本；
3. 用户进入项目聊天时，无需重复说明项目 Goal 和历史决策；
4. 上传文档或同步 Agent 记录后，可以自动推荐项目并进入项目 L3；
5. L3 能体现多来源变化，而不是简单拼接会议摘要；
6. 用户通过聊天创建的任务可在项目中查看并由 App 获取提醒；
7. L4 判断必须展示证据并经过用户确认，否定后不再以事实注入回答；
8. 任一重要洞察可以回到原始录音、聊天或文档；
9. 系统无法确定时明确表达不确定性，不用推测填补个人事实；
10. 用户持续使用和修正后，相同问题的后续回答能够体现已确认的新认知。
